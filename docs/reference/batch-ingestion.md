# 07. 데이터 수집 및 배치 전략

## 배치가 필요한 이유

외부 공공 API를 사용자 요청마다 실시간 호출하면 응답 속도가 느려지고 장애에 취약합니다. 실거래가와 안전시설 데이터는 백엔드 배치 작업으로 미리 수집해 DB에 저장합니다.

---

## 실거래가 수집

| 항목 | 내용 |
| --- | --- |
| 대상 API | 국토교통부 실거래가 공공 API 8종 (전월세 4종 + 매매 4종) |
| 주택 유형 | 단독/다가구, 연립/다세대, 오피스텔, 아파트 |
| 수집 주기 | 매일 새벽 배치 |
| 저장 테이블 | `transaction_history` |
| 활용 | 매물 상세 시세 비교, 시세 분석, 가격 점수 계산 |

### 실거래가 매칭 기준 (MVP)

| 우선순위 | 기준 | 설명 |
| --- | --- | --- |
| 1 | 법정동 코드 | 매물의 `legal_dong_code`와 같은 법정동의 거래를 우선 조회 |
| 2 | 주택 유형 | 매물의 `property_type`과 같은 유형의 거래를 우선 사용 |
| 3 | 전용면적 | 매물 `area_m2` 기준 ±10㎡ 범위를 유사 거래로 판단 |
| 4 | 거래 기간 | 기본 최근 3년 거래 사용 |
| 5 | 거래 종류 | 전세는 보증금 중심, 월세는 보증금과 월세를 함께 비교 |

## 안전시설 수집

| 항목 | 내용 |
| --- | --- |
| 대상 데이터 | CCTV, 비상벨, 보안등, 치안시설 (파출소·지구대) |
| 수집 주기 | 주 1회 |
| 저장 테이블 | `safety_facility` |
| 활용 | 지도 오버레이, 안전 요약, 안전 점수 계산 |

범죄주의구간 WMS는 DB 저장 없이 프론트 지도 레이어에서 직접 렌더링합니다.

---

## 점수 계산 배치

안전시설과 실거래가가 저장된 뒤 매물별 점수를 계산합니다.

```
매물 기준 반경 검색
→ CCTV/비상벨/보안등/치안시설 개수 계산
→ 안전 점수 계산 (항목별 0~100 정규화 → 가중 평균)
→ 실거래가 기반 가격 점수 계산
→ property_score_stat 저장
```

### 안전 점수 가중치

| 항목 | 반경 | 가중치 |
| --- | --- | --- |
| CCTV | 300m | 30% |
| 안전비상벨 | 300m | 25% |
| 보안등/방범등 | 300m | 25% |
| 치안시설 (파출소·지구대) | 500m | 20% |

## 뉴스 인덱싱 (1.5차)

| 항목 | 내용 |
| --- | --- |
| 대상 | 딥서치 API 기반 부동산 정책·시장 동향 뉴스 |
| 처리 | 청킹 → 임베딩 → pgvector 저장 |
| 수집 주기 | 주기적 재인덱싱 (최신성 유지) |
| 활용 | F-4 뉴스 RAG 응답 생성 |

## 운영 방식

MVP에서는 자동 스케줄보다 수동 실행 가능한 Spring Scheduler 또는 CommandLineRunner로 먼저 구현합니다.
