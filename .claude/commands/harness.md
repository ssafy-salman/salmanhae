# /harness — 원스톱 구현 실행

살만해 프로젝트의 하네스 실행 스킬입니다. 다음 순서로 진행합니다.

## 실행 흐름

### 1단계: 문서 읽기
다음 파일을 전부 읽어 프로젝트 컨텍스트를 파악합니다.
- `CLAUDE.md` — 기술 스택, 아키텍처 규칙, 개발 프로세스
- `docs/01_PRD.md` — 핵심 기능, MVP 제외 사항
- `docs/02_ARCHITECTURE.md` — 디렉토리 구조, 패턴, 데이터 흐름
- `docs/03_ADR.md` — 기술 결정 및 트레이드오프
- `docs/04_UI_GUIDE.md` — 색상, 컴포넌트 패턴, 안티패턴
- `docs/11_ROADMAP.md` — 개발 순서 (단계별 상세)

### 2단계: 사용자와 논의
문서를 읽은 뒤 사용자에게 다음을 확인합니다.
- 이번에 구현할 기능 범위 (어떤 단계/Feature인지)
- 특별히 주의할 제약 조건이나 우선순위
- 의존성이 있는 선행 작업이 완료되었는지

### 3단계: Phase 계획 수립
구현 계획을 Phase로 나눕니다.
- 각 Phase는 독립적으로 실행 가능한 단위
- Phase 1개당 예상 작업 시간: 3~10분
- Phase 수: 5~7개 권장 (너무 많거나 적으면 재조정)
- 각 Phase 파일에는 다음을 포함:
  - 작업 목표 (1~2줄)
  - 수정/생성할 파일 목록
  - 완료 조건 (어떤 상태이면 완료인지)
  - 의존 Phase (없으면 생략)

사용자에게 Phase 목록을 먼저 보여주고 승인을 받습니다.

### 4단계: Phase 파일 생성
승인 후 `phases/{task-name}/` 폴더를 만들고 Phase 파일을 생성합니다.

파일 네이밍: `phase{N}-{slug}.md`  
예: `phases/mvp/phase1-project-init.md`

Phase 파일 형식:
```markdown
# Phase {N}: {제목}

## 목표
{1~2줄 설명}

## 작업 파일
- `{파일 경로}` — {무엇을 할지}

## 완료 조건
- [ ] {체크 항목 1}
- [ ] {체크 항목 2}

## 아키텍처 규칙 체크
- CLAUDE.md의 CRITICAL 규칙 중 이 Phase와 관련된 것:
  - {관련 규칙 열거}

## 구현 지시
{Claude가 수행할 상세 구현 내용}
```

### 5단계: execute.py 실행 (선택)
Phase 파일 생성 완료 후 사용자에게 확인:
> "Phase 파일이 생성되었습니다. `python3 scripts/execute.py {task-name}` 으로 자동 실행할까요? 아니면 수동으로 진행하시겠어요?"

자동 실행을 원하면 Bash로 실행합니다.

## 준수 사항

- MVP 제외 사항(`docs/01_PRD.md`)에 있는 기능은 Phase에 포함하지 않는다.
- 각 Phase는 CLAUDE.md의 CRITICAL 규칙을 명시적으로 확인한다.
- API 키 관련 작업이 있으면 반드시 환경변수 처리를 Phase에 포함한다.
- 테스트 작성을 별도 Phase로 두지 않고 구현 Phase 내에 포함한다 (TDD).
